# Echo 本地视频中心 PRD

## 1. 文档概览

### 1.1 产品名
Echo

### 1.2 产品定位
一款 **Desktop-first、本地优先** 的视频中心，主要运行在用户自己的电脑上。Echo 用本地处理链完成视频获取、元数据下载、缩略图保存、音频提取、语音转写、字幕翻译、字幕生成与最终视频输出，然后把已完成内容组织成可直接播放的本地媒体库。

产品的核心价值从“任务处理台”升级为“播放优先的本地视频站”：用户打开应用后首先看到的是可观看的视频卡片、缩略图、标题、来源、时长、字幕状态与简介；导入文件、URL 下载和处理队列是辅助入口，而不是首页中心。

Echo 借鉴 YouTube / Bilibili 的浏览和进入播放方式，但不做公共视频网站、账号社区或云端分发。所有视频、元数据、字幕和缩略图都优先保存在本地，播放也围绕用户自己的本地资产展开。

### 1.3 核心问题
大量视频内容仍然以用户无法直接消费的语言存在。现有工具常见问题包括：
- 依赖云端或 SaaS，不适合私有素材、长视频、重复使用。
- 下载、处理、播放、媒体管理揉在一起，边界模糊，维护成本高。
- 工具把任务详情放在首页，用户每次都先面对处理过程，而不是面对可观看内容。
- 只生成文件但不组织成媒体库，用户很难快速找回、继续观看或复习字幕。
- 任务失败后难以恢复，只能从头重跑。
- 下载工具只保留视频文件，没有充分利用 yt-dlp 可获得的标题、缩略图、频道、简介、章节、发布时间等元数据。

### 1.4 产品目标
首版让用户能够：
1. 导入本地视频，或输入在线视频 URL。
2. 在本地完成视频下载、元数据获取、语音转写、字幕翻译与字幕生成。
3. 在首页以视频卡片方式浏览已完成内容，包含缩略图、标题、来源、时长、字幕状态、描述和清晰的播放入口。
4. 点击任意已完成视频后直接进入播放并自动开始播放。
5. 在辅助区域查看处理队列、失败原因、重试入口和产物位置。
6. 在 CLI 中以同等核心能力完成自动化批处理。

### 1.5 成功标准
MVP 发布后，产品应满足：
- 新用户可在 10 分钟内完成首次成功处理。
- 常见失败可定位、可重试、可从中间阶段恢复。
- 同一输入在 Desktop 与 CLI 下遵循同一任务模型、同一流水线、同一产物规范。
- 产物目录可被用户长期保留，并能被外部媒体工具直接读取。
- 处理完成后，视频自动进入本地视频库，用户不需要先理解任务详情。
- 点击视频卡片直接进入播放，不再二次点击播放按钮。
- 后续新增能力时，不需要推翻首版核心架构。

## 2. 目标用户与核心使用场景

### 2.1 目标用户

#### A. 本地视频观看与学习用户
- 有大量外语视频需要理解。
- 关注隐私、成本和离线处理。
- 希望像刷视频网站一样浏览自己的本地视频，但保留双语字幕、逐句复习和离线资产控制。

#### B. 媒体收藏与整理用户
- 使用 Jellyfin、Plex、Kodi 或本地 NAS 维护个人媒体库。
- 希望生成的字幕和视频能直接放进已有目录结构中长期保存。
- 不需要本产品替代媒体中心，只需要稳定地产出规范文件。

#### C. 技术型用户 / 自动化用户（后续阶段）
- 未来可能通过 CLI 处理批量任务。
- 有批量处理、定时脚本、规则化归档需求。
- 这一人群会影响架构设计，但不要求在 MVP 中直接交付 CLI。

### 2.2 核心使用场景

#### 场景 1：单个本地视频处理
用户拖入一个本地 MP4 文件，选择目标语言和输出目录，点击开始。系统自动生成原文字幕、译文字幕、双语字幕与最终视频。

#### 场景 2：在线视频转本地资产
用户粘贴一个在线视频链接。系统通过 yt-dlp 获取视频、标题、频道、缩略图、简介、章节、发布时间等元数据，再走同一处理链，最终进入本地视频库。

#### 场景 3：批量处理
用户在 WSL 中通过 CLI 扫描一个文件夹或传入多个 URL，批量生成字幕与视频，适合夜间跑任务或脚本化归档。

#### 场景 4：失败恢复
长视频转写到一半因显存不足或断电失败。用户重新打开应用后，能看到失败节点、已有产物，并从最近一个成功阶段继续，而不是全量重跑。

#### 场景 5：媒体库旁路协作
用户将输出目录设置为某个 Jellyfin / Plex / Kodi 已监听的文件夹。处理完成后，外部工具自行扫描文件；本产品不调用其 API，也不承担同步与展示责任。

#### 场景 6：像视频网站一样进入播放
用户在首页看到“4 CAVALIERS at 3 KNICKS FULL GAME 1 HIGHLIGHTS”的缩略图、时长、来源、字幕状态和简介。点击卡片后立即进入播放页，视频自动开始播放，右侧同步字幕列表可用于逐句跳转、当前句重播和单句循环。

## 3. 产品边界与非目标

### 3.1 产品边界
本产品负责：
- 输入获取：本地文件导入、URL 下载调度。
- 元数据获取：标题、频道 / 作者、简介、缩略图、时长、发布时间、章节、来源 URL。
- 视频处理：音频提取、转写、翻译、字幕生成、封装或烧录字幕。
- 本地视频库：把已完成内容组织成可浏览、可搜索、可筛选的视频卡片。
- 播放页：围绕本地视频与字幕产物进行播放、逐句跳转、回放、循环和字幕同步浏览。
- 任务编排：进度、状态、日志、失败恢复、重试。
- 本地资产输出：视频文件、字幕文件、元数据与日志。
- 跨端一致性：Desktop App 与 CLI 复用同一核心能力。

本产品不负责：
- 公共视频网站、推荐流、账号体系、评论弹幕社区。
- 外部媒体系统同步。
- 云端账户体系。
- 局域网服务化播放。

### 3.2 非目标
首版明确不做：
- 自建媒体中心。
- Jellyfin / Plex / Kodi API 级集成。
- 局域网播放服务。
- 独立片库、收藏夹、评分、海报墙、播放历史同步。
- TTS 配音。
- 说话人分离。
- 复杂剪辑、时间线编辑。
- 多人协作。
- 云端同步与 SaaS 化。
- 面向商业发行的一整套授权、计费、团队管理能力。

### 3.3 设计原则
1. **Desktop-first，不是 server-first。**
2. **处理核心与交互层分离。** Desktop 和 CLI 只是不同入口。
3. **下载能力与处理能力解耦。** 视频从哪里来，不应污染后续处理逻辑。
4. **播放优先。** 首页首先服务“找视频、点进去、马上看”，处理任务退到辅助区域。
5. **元数据是一等资产。** yt-dlp 获取到的缩略图、标题、频道、简介、章节和发布时间必须进入 manifest 和 UI。
6. **文件资产优先。** 结果应可长期保存、迁移、再利用。
7. **任务可恢复。** 每一步都有可验证输入、输出与状态。
8. **模块边界清晰。** 避免“所有功能共用一锅状态”的旧式耦合。
9. **外部工具友好，但不依赖外部工具。**

## 4. 用户主流程

### 4.1 主流程

```text
选择输入
  ├─ 本地视频文件
  └─ 在线视频 URL
        ↓
选择任务参数
  ├─ 输出目录
  ├─ 源语言 / 自动识别
  ├─ 目标语言
  ├─ 下载方式
  ├─ ASR 模型
  └─ 字幕输出形式
        ↓
创建任务
        ↓
获取输入资产
        ↓
探测媒体信息
        ↓
提取音频
        ↓
本地 ASR 转写
        ↓
字幕分段与校正
        ↓
字幕翻译
        ↓
生成字幕文件
        ↓
生成最终视频文件
        ↓
写入任务清单与产物索引
        ↓
进入本地视频库
        ↓
用户点击视频卡片直接播放 / 查看产物 / 打开输出目录 / 继续批处理
```

### 4.2 关键状态
- 等待中
- 下载中
- 处理中
- 已暂停
- 已失败
- 可恢复
- 已完成
- 已取消

### 4.3 失败体验要求
- 用户必须能知道：
  - 哪一步失败。
  - 失败原因的大类。
  - 是否可重试。
  - 重试后会从哪里继续。
- 对常见错误提供可操作提示，例如：
  - GPU 显存不足。
  - FFmpeg 不可用。
  - 下载器配置错误。
  - 模型缺失。
  - 输出目录不可写。

## 5. Desktop App 与 CLI 的角色分工

### 5.1 共同点
Desktop App 与 CLI 必须共享：
- 核心引擎。
- 任务模型。
- 流水线定义。
- 产物规范。
- 配置模型。
- 错误码与恢复机制。

它们的关系应更像：

```text
           ┌──────────────┐
           │ Core Engine   │
           │ Pipeline      │
           │ Task Model    │
           │ Artifact Spec │
           └──────┬───────┘
                  │
        ┌─────────┴─────────┐
        │                   │
┌───────▼────────┐  ┌───────▼───────┐
│ Win11 Desktop  │  │ WSL Ubuntu CLI │
│ GUI / UX       │  │ Automation     │
└────────────────┘  └───────────────┘
```

### 5.2 Desktop App 负责
- 面向普通用户的导入、配置、进度查看。
- 任务卡片、历史任务、失败重试。
- 模型与下载器设置引导。
- 输出目录选择与打开。
- GPU / 依赖项健康检查。
- 对复杂日志做“人类可读”摘要。

### 5.3 CLI 负责
- 批处理。
- 脚本化。
- 与其他本地自动化流程衔接。
- 明确的命令参数、退出码、结构化输出。
- 无 GUI 环境下的完整能力覆盖。

### 5.4 明确不接受的分叉
- Desktop 走一套私有流程、CLI 走另一套脚本流程。
- 两端生成不同目录结构或不同文件命名。
- 两端对任务状态与错误类型定义不一致。

## 6. 功能需求与优先级

### 6.1 P0：MVP 必需

| 模块 | 功能 | 说明 |
|---|---|---|
| 输入 | 导入本地视频 | 支持常见视频格式，至少 MP4 / MKV / MOV |
| 输入 | 输入在线视频 URL | 支持由本地自托管 Cobalt、yt-dlp 等下载器获取 |
| 下载 | 下载适配层 | 下载器插件化，处理链不感知具体下载器 |
| 媒体处理 | 提取音频 | 基于 FFmpeg |
| ASR | 本地语音转写 | 支持本地 GPU 模型，首版可从单一默认模型起步 |
| 翻译 | 字幕翻译 | 支持目标语言选择 |
| 字幕 | 原文字幕 | 独立文件输出 |
| 字幕 | 译文字幕 | 独立文件输出 |
| 字幕 | 双语字幕 | 独立文件输出 |
| 输出 | 最终视频文件 | 至少支持 remux / mux 字幕；是否默认烧录可配置 |
| 任务 | 进度展示 | 当前阶段、总进度、耗时、日志摘要 |
| 任务 | 失败原因 | 显示错误阶段与错误描述 |
| 任务 | 重试 | 可对失败任务重试 |
| 任务 | 历史任务 | 可查看已完成、失败、取消任务 |
| 配置 | 输出目录 | 可指定任意本地目录 |
| 配置 | 模型 / 语言 / 下载方式 | 基础可配置 |
| CLI | 同等核心流程 | 本地文件和 URL 均可处理 |

### 6.2 P1：强烈建议尽早纳入

| 模块 | 功能 | 说明 |
|---|---|---|
| 任务 | 阶段级恢复 | 从最近成功阶段继续 |
| 任务 | 任务清单文件 | 每次运行均生成 manifest |
| 配置 | 预设配置 | 如“快速”“高质量”“仅字幕” |
| 输出 | 多种字幕格式 | 至少 SRT，后续可加 ASS / VTT |
| 输出 | 仅生成字幕模式 | 不生成最终视频，节省时间 |
| 健康检查 | 依赖检测 | FFmpeg、GPU、模型、下载器可用性 |
| CLI | 批量输入 | 目录、清单文件、glob |
| 日志 | 结构化日志 | 便于排错和未来自动化分析 |

### 6.3 P2：后续演进

| 模块 | 功能 | 说明 |
|---|---|---|
| ASR | 多模型管理 | 不同速度 / 精度模型 |
| 字幕 | 字幕编辑器 | 轻量校正，不做复杂 NLE |
| 翻译 | 多翻译引擎 | 本地 / 云端 / 第三方可切换 |
| 输出 | 多轨封装 | 多语言字幕轨、章节信息 |
| 下载 | 更多下载器适配 | 插件式扩展 |
| 批处理 | 队列策略 | 并发、优先级、夜间任务 |
| 质量 | 术语表 / 风格表 | 提升字幕一致性 |

## 7. 关键页面与交互设计

### 7.1 首页 / 本地视频库
目标：让用户像使用视频网站一样浏览本地视频，优先看到可播放内容，而不是任务详情。

核心元素：
- 顶部搜索：标题、来源、简介、字幕内容。
- 辅助动作：导入视频、下载器、设置。
- 左侧导航：首页、媒体库、处理中、下载中、播放列表。
- 已完成视频网格：缩略图、时长、标题、来源、字幕状态、简介、添加时间、清晰度。
- 处理队列条：显示仍在下载 / 转写 / 翻译的任务，但不占据首页核心。

交互原则：
- 点击已完成视频卡片直接进入播放页并自动开始播放。
- 任务详情不再常驻首页；只有处理队列、失败重试和产物位置作为辅助信息出现。
- URL 下载入口可以展开，但默认不抢占媒体库空间。

### 7.2 新建任务页
核心字段：
- 输入来源。
- 输出目录。
- 源语言：自动识别 / 手动指定。
- 目标语言。
- 下载方式。
- 输出选项：原文字幕、译文字幕、双语字幕、最终视频。
- 高级选项折叠区：模型、精度、字幕格式、是否仅字幕。

交互原则：
- 默认配置必须能直接开跑。
- 高级设置不能阻塞普通用户。
- 对依赖缺失即时提示，而不是等任务失败后才暴露。

### 7.3 处理队列页
应展示：
- 当前阶段。
- 阶段时间线。
- GPU / CPU / 磁盘占用摘要。
- 当前日志摘要。
- 产物列表。
- 失败原因与建议动作。
- 重试、暂停、删除、打开输出目录。

已完成任务不再进入任务详情页，而是进入视频库卡片和播放页。

### 7.4 播放页
应展示：
- 本地视频画面。
- 生成的字幕轨，默认优先双语字幕。
- 同步字幕列表。
- 播放 / 暂停、上一句、下一句、重播当前句。
- 单句循环。
- 播放速度。
- 全屏播放。
- 打开本地位置 / 复制路径等资产动作。

交互原则：
- 播放页只从已完成视频进入。
- 点击卡片后自动播放；如系统策略阻止自动播放，仍停留在播放页并显示可用播放按钮。
- 字幕列表和视频进度必须保持同步，当前句要明显可见。

### 7.5 历史任务页
支持：
- 按状态筛选。
- 搜索标题 / 输入源。
- 查看产物。
- 从历史任务复制配置重新运行。
- 清理中间文件但保留最终资产。

### 7.6 设置页
分类建议：
- 通用：默认输出目录、文件命名规则。
- 模型：ASR 模型、缓存目录、GPU 策略。
- 下载：Cobalt、yt-dlp、默认下载器、代理。
- 输出：默认字幕格式、封装策略。
- 高级：临时目录、日志级别、并发策略。

## 8. 文件产物与输出目录规范

### 8.1 原则
输出目录不是一次性缓存，而是用户资产的一部分。规范必须：
- 可读。
- 可迁移。
- 可恢复。
- 对外部媒体工具友好。
- 不依赖数据库也能理解基本内容。

### 8.2 推荐目录结构

```text
<output-root>/
  <asset-slug>/
    source/
      original.mp4
      source.json
    subtitles/
      <asset-slug>.source.srt
      <asset-slug>.translated.<lang>.srt
      <asset-slug>.bilingual.<lang>.srt
    video/
      <asset-slug>.final.<lang>.mkv
    transcripts/
      transcript.json
      segments.json
    logs/
      task.log
      errors.log
    manifest.json
```

### 8.3 文件说明
- `source/original.*`：输入视频的本地副本或下载结果。
- `source/source.json`：输入来源、URL、下载器、hash、媒体信息。
- `subtitles/*.srt`：长期保存字幕资产。
- `video/*.final.*`：最终视频文件。
- `transcripts/*.json`：机器可读中间结果，供恢复和后续加工。
- `manifest.json`：本次任务的总索引，记录配置、阶段、版本、产物、hash。

### 8.4 命名规则建议
- 使用稳定 slug，避免完全依赖原始标题。
- 文件名应携带语义，而非依赖目录猜测。
- 语言码采用统一标准，如 `zh-CN`、`en`。
- 对媒体中心友好的输出可附带同名字幕副本，例如：
  - `Movie Name (2024).mkv`
  - `Movie Name (2024).zh-CN.srt`
  - `Movie Name (2024).en.zh-CN.srt`

### 8.5 临时文件与长期资产分离
- 中间缓存与可丢弃临时文件应放在独立 temp / cache 区域。
- 用户清理缓存时，不应误删最终视频、字幕与 manifest。

## 9. 配置项与任务恢复机制

### 9.1 配置项

#### 全局配置
- 默认输出目录。
- 默认临时目录。
- 默认 ASR 模型。
- 默认源语言策略。
- 默认目标语言。
- 默认下载方式。
- 默认字幕格式。
- 是否保留中间文件。
- GPU 使用策略。

#### 单任务配置
- 输入源。
- 输出目录。
- 下载器与下载参数。
- ASR 模型。
- 源语言 / 目标语言。
- 需要生成的产物类型。
- 是否仅生成字幕。
- 是否覆盖已有产物。

### 9.2 配置优先级
建议采用：

```text
CLI 显式参数 / Desktop 单任务配置
    > 用户配置文件
    > 系统默认值
```

### 9.3 任务模型
每个任务应具备：
- `task_id`
- `input_fingerprint`
- `config_snapshot`
- `pipeline_version`
- `stage_states`
- `artifacts`
- `created_at / updated_at`
- `resume_policy`

### 9.4 阶段模型
每个阶段至少记录：
- 阶段名。
- 输入依赖。
- 输出产物。
- 开始 / 结束时间。
- 状态。
- 错误摘要。
- 可否重试。
- 产物 hash 或存在性校验。

### 9.5 恢复规则
- 阶段输出存在且校验通过，可跳过。
- 上游输入变化，则下游阶段失效。
- 用户显式“强制重跑某阶段”时，该阶段及下游阶段重新执行。
- 如果下载完成、转写失败，应从转写阶段恢复，而不是重新下载。
- 如果字幕翻译策略变化，仅翻译与下游字幕 / 视频阶段重跑。

### 9.6 恢复体验
Desktop：
- 在任务详情页显示“可从转写阶段继续”。
- 对用户解释哪些结果会复用、哪些会重算。

CLI：
- 提供如 `resume <task-id>`、`rerun --from translate`、`--force subtitle` 等能力。

## 10. 技术约束与架构原则

### 10.1 平台约束
- 主平台：Windows 11 Desktop。
- 辅助平台：WSL Ubuntu CLI。
- 本地 GPU 为首选运行路径。
- 大文件、长任务、断点恢复是默认场景，不是边角场景。

### 10.2 推荐架构分层

```text
UI Layer
  ├─ Win11 Desktop App
  └─ CLI

Application Layer
  ├─ Task Service
  ├─ Config Service
  ├─ Artifact Service
  └─ Recovery Service

Domain / Core Layer
  ├─ Pipeline Engine
  ├─ Stage Definitions
  ├─ Task Model
  └─ Artifact Spec

Adapter Layer
  ├─ Downloader Adapters (Cobalt / yt-dlp / local import)
  ├─ ASR Adapter
  ├─ Translation Adapter
  ├─ FFmpeg Adapter
  └─ Storage Adapter
```

### 10.3 架构原则
- 核心流水线不依赖 GUI。
- 下载器、ASR、翻译器、封装器均应适配器化。
- 任务状态以本地可持久化数据为准，不依赖进程内内存。
- 产物规范应版本化。
- 配置应快照化，保证历史任务可解释。
- 任何阶段都应具备独立执行与独立测试能力。
- 日志要面向人类与机器两种消费者。
- 不让媒体管理逻辑侵入核心域。

### 10.4 对旧项目的直接纠偏
- 不再同时把“CLI 工具、API 服务、任务编排平台、媒体处理流程”揉进一个产品定义里。
- 不把 TTS、播放、媒体中心、云服务一起塞进首版。
- 不把“运行一次的目录”当成“长期资产目录”。
- 不让输入获取方式决定后续流水线形态。

## 11. MVP 范围

### 11.1 MVP 必须交付
- Win11 Desktop App。
- 本地视频导入。
- 在线 URL 输入与下载。
- 音频提取。
- 本地 ASR。

### 11.2 MVP 可接受的简化
- 先只支持少量主流视频格式。
- 先只内置一个默认 ASR 模型。
- 先只支持 SRT。
- 翻译引擎先支持一种默认方案。
- 最终视频先支持一种推荐封装方式。
- Desktop UI 先以清晰可靠为主，不追求媒体库级审美。

## 13. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| Windows 与 WSL 环境差异 | 核心能力复用受阻 | 早期就定义跨端抽象层与统一测试样例 |
| 本地 GPU 生态复杂 | 用户首跑失败率高 | 健康检查、默认配置、依赖引导、错误诊断 |
| 下载源变化快 | URL 下载不稳定 | 下载器适配层、可切换实现、与处理链隔离 |
| 长任务失败成本高 | 用户信任下降 | 阶段持久化、manifest、断点恢复 |
| 产物命名混乱 | 难以长期管理 | 一开始固定规范并版本化 |
| 功能膨胀 | 再次回到“大杂烩” | 以边界清单约束路线图 |
| 翻译质量不稳定 | 成品体验波动 | 支持后续引擎替换、保留机器可读中间产物 |
| 磁盘占用高 | 用户误删或放弃 | 临时资产与长期资产分离、清理策略透明 |

## 14. 建议的首版验收标准

1. 用户可通过 Desktop 处理一个本地视频并得到：
   - 原文字幕。
   - 译文字幕。
   - 双语字幕。
   - 最终视频。
   - manifest。
2. 用户可通过 URL 完成同样流程。
3. 用户可在 CLI 中以等价参数完成同样流程。
4. 任一阶段失败后，系统可展示失败阶段并在重试时复用已有上游产物。
5. 输出目录迁移到另一台机器后，用户仍可通过文件本身理解任务基本信息。
6. 若输出目录被 Jellyfin / Plex / Kodi 监听，外部系统可自行读取兼容文件；本产品不依赖它们即可完整工作。

## 15. 一句话总结
Echo v0.2 不是公共视频网站，而是放在用户桌面上的本地视频中心：用视频网站式的浏览和播放效率，管理用户自己的本地视频、元数据、字幕和学习资产。
